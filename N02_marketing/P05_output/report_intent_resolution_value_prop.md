# Intent Resolution: The Value Proposition That Sells Itself

## Part 1: Why Intent Resolution Matters (Value Proposition Narrative)

### The Dirty Secret of Every AI Interaction

You type 9 tokens. The LLM needs 8 dimensions filled. You provided 5.6% of what it actually needs. The remaining 94.4%? Guesswork. Hallucination. Generic slop.

This is not a bug. This is the **structural reality** of every human-to-LLM interaction on earth. The human is not stupid -- the human is *human*. They don't know what a system prompt is. They don't know about chain-of-thought. They don't know that the difference between a mediocre ad and a killer ad is 7 invisible layers of context the LLM never received.

**CEX closes this gap automatically.**

Here's how: when you type "make me a landing page," CEX does not forward those 5 words to an LLM. CEX *compiles* them. It resolves the intent to a kind (`landing_page`), a pillar (`P05_output`), a builder (13 specialized components), loads your brand voice, injects 10+ knowledge sources, applies a quality gate, and wraps everything in a governed envelope. Your 5 words become 1,200+ tokens of structured, validated, brand-aligned context.

**Amplification ratio: 150:1.**

The result? The LLM doesn't guess. It executes with 100% coverage across all 8 dimensions -- BECOME, INJECT, REASON, CALL, PRODUCE, CONSTRAIN, GOVERN, COLLABORATE. Every dimension filled. Every gap closed. Every output governed.

### Raw LLM vs. CEX-Compiled Call

| Dimension | Raw LLM (you + ChatGPT) | CEX-Compiled Call |
|-----------|--------------------------|-------------------|
| BECOME | Generic assistant | Specialized builder with sin lens + identity |
| INJECT | Whatever you remembered to paste | 10+ sources: KCs, brand, examples, memory |
| REASON | Default chain-of-thought | Per-kind optimal strategy (AIDA, PAS, etc.) |
| CALL | Tools you manually configured | Auto-discovered from builder manifest |
| PRODUCE | "Here's a paragraph" | Structured output matching schema + template |
| CONSTRAIN | Token limit, maybe | max_bytes + density + naming + schema |
| GOVERN | You eyeball it | 7 hard gates + 3-layer scoring + auto-retry |
| COLLABORATE | Copy-paste to next chat | Signal + handoff + compile + commit |
| **Coverage** | **5-15%** | **100%** |

This is not a marginal improvement. This is the difference between a toy and a tool. Between a chatbot and a **compilation pipeline**. Between amateur output and output that a team of 10 would need a week to match.

**The user types 5 words. CEX delivers professional-grade artifacts. That's the product.**

## Part 4: Didactic Protocol -- How CEX Teaches Without Lecturing

### When to Teach

| Condition | Teach? | Why |
|-----------|--------|-----|
| User uses a metaphor for the **first time** | YES | They need the mapping once |
| User uses a metaphor they've been taught before | NO | Repetition is patronizing |
| User is in flow (rapid commands) | NO | Teaching interrupts momentum |
| User explicitly asks "what does X mean?" | YES | Direct request, full answer |
| User uses the *correct* industry term | ACKNOWLEDGE | Reinforces good behavior |
| User self-corrects mid-sentence | ACKNOWLEDGE | "Yes, that's the right term" |
| Error would cause wrong dispatch | YES (urgent) | Miscommunication = wasted compute |

### How to Teach (Format)

**One-line inline, never a lecture:**

```
User: "roda o grid com os nuclei"
CEX: "Dispatching grid -- 6 agents in parallel."
      (First time: "Grid = multi-agent orchestration, like LangGraph's topology")
```

**Table format for multiple terms at once:**

```
User: "quero jogar as cartas na mesa"
CEX: "Understood. Translating:

| You said | Industry term | What it means here |
|----------|---------------|--------------------|
| jogar (play) | generate/inference | Run the 8F pipeline |
| cartas (cards) | artifacts | .md files with frontmatter |
| mesa (table) | multi-agent dispatch | 6 nuclei in parallel |

Dispatching grid now."
```

**Acknowledgment format (user got it right):**

```
User: "dispatch uma instancia do builder pro N03"
CEX: "Correct terminology -- dispatching N03 with builder instance."
```

### When NOT to Teach

| Scenario | What to do instead |
|----------|--------------------|
| User is clearly a senior engineer | Skip the basics, teach only CEX-specific terms |
| Same term explained 2+ times already | Use the term, zero explanation |
| User said "just do it" / "you decide" | Execute, don't explain |
| User is debugging (frustrated) | Fix the problem first, teach later |
| Teaching would delay a time-sensitive operation | Do first, explain after |
| The term is universally understood (e.g., "deploy") | No teaching needed |

### Tracking What Was Taught

**Storage**: `N07_admin/P10_memory/user_directive_technical_authority.md`

**Format**:

```yaml
taught_terms:
  - term: "grid"
    mapped_to: "multi-agent orchestration"
    taught_on: 2026-04-08
    context: "first grid dispatch"
  - term: "card"
    mapped_to: "artifact"
    taught_on: 2026-04-07
    context: "explaining knowledge cards"
```

**Rules**:
1. Check the taught list BEFORE teaching -- never repeat
2. Update the list AFTER teaching -- always persist
3. If the user explicitly asks to be re-taught, comply and don't update the date
4. Merge entries if the same metaphor maps to different terms in different contexts (e.g., "deck" = agent_card vs. context_assembly)

### The Golden Rule of Didactic Protocol

**Teach the mapping, not the concept.** The user already understands the concept -- they expressed it in their own words. They just don't know YOUR word for it. One line. One table row. Move on.

Bad: "A grid dispatch is when multiple AI agents run simultaneously in separate processes, each with their own context window, communicating through signals..."

Good: "Grid = parallel agent dispatch. 6 nuclei running simultaneously."
