## Summary

The 13-ISO family for `dual_loop_architecture` was generated by qwen3:8b and consistently mis-framed the kind as **classical control systems** (PID, actuators, Lyapunov, ISO 26262, robotics) rather than the intended target: **LLM agent dual-loop / System-1 vs System-2 / fast-slow inference** (Reflexion, OODA, Skeleton-of-Thought, Kahneman).

The robotics framing is not wrong in the abstract -- dual-loop IS a control-theory concept -- but every artifact in CEX must land in the LLM-agent context (P08 architecture for agents, not for aerospace). No ISO mentions: tick_ms, cadence_s, model tiers (fast vs reasoning), memory bridge, plan slot, Reflexion, OODA, Sonnet/Opus split, o1/extended-thinking. `bld_architecture` went fully off-topic and enumerated a crypto exchange (Transaction Processor, API Gateway, Compliance Checker).

Secondary failures: low frontmatter density (<=13 fields vs gold's 19+), no `keywords`/`references`, placeholder hooks (`null`, `null`, `null`), generic tool lists (cex_optimizer.py does not exist), and missing dual-loop specifics (feedback vs critic vs regret, async vs sync bridge, tick/cadence ratio).

## Top Issues (Ranked)

1. **Wrong frame**: robotics/PID instead of LLM agents. Inner loop should be "small fast LLM emitting actions", not "PID controller for actuators". Outer should be "reasoning model revising plan", not "Model Predictive Control with Lyapunov".
2. **No cadence vocabulary**: no `tick_ms`, no `cadence_s`, no ratio constraint, no early-replan-on-regret. These are the load-bearing numeric knobs of any dual-loop spec.
3. **bld_architecture hallucinated a trading platform**: "Transaction Processor", "API Gateway", "Compliance Checker", "Legal owner". Zero connection to dual_loop_architecture. Likely qwen3:8b confused "CEX" (our project) with "crypto exchange".
4. **bld_quality_gate has wrong ID pattern**: `P08-XX-XXX` contradicts `bld_schema` which correctly says `^p08_dl_[a-zA-Z0-9_]+\.md$`. Hard gate H02 would never pass.
5. **bld_tools references non-existent tools**: `cex_optimizer.py`, `cex_validator.py`, `val_security_scanner.py` -- none in `_tools/`. Should point to real tools (`cex_compile`, `cex_score`, `cex_doctor`, `cex_retriever`) + domain-specific runtime tools (tick_scheduler, memory_bridge).
6. **bld_config body is 13 lines**: three `null` hooks, no cadence knobs, no model-tier defaults, no bridge topics. Dense-frontmatter-thin-body is the opposite of what we want.
7. **Frontmatter density**: most ISOs have 10-12 fields. Gold reference has 19+ (keywords, triggers, capabilities, references, parent, etc.).
8. **bld_collaboration** treats "Inner Loop" and "Outer Loop" as external builders in the Receives/Produces tables -- category error. The loops are INTERNAL to the artifact this builder produces, not crew peers.
9. **bld_examples golden case is OK** (autonomous vehicle) but still robotics-flavored. Missing an LLM-agent golden (e.g., Sonnet-inner + Opus-outer coding agent, or Reflexion-style self-critique).
10. **bld_memory is generic** -- "boundary conditions", "failure modes" could apply to any kind. No mention of trajectory buffer, plan slot staleness, bridge schema drift.
