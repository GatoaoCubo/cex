# Summary
The most reliable way to execute N04 knowledge work was to read the builder contract, kind KC, pillar schema, and nearby nucleus examples before drafting.
That sequence kept the artifacts inside the correct boundaries while still allowing the Knowledge Gluttony lens to make them broad, connected, and information-dense.
The practical win was less rewrite pressure: structure decisions stabilized early, so later passes focused on depth rather than repair.

## Pattern
1. Load the pillar schema first so kind boundaries and naming expectations are explicit before any prose appears.
2. Read the kind builder manifest and instruction before drafting so section shape follows an existing contract.
3. Pull the kind KC and local N04 examples next to anchor vocabulary in repository-native terms.
4. Tie each artifact back to N04 identity, memory posture, and retrieval goals so the Knowledge Gluttony lens stays domain-focused.
5. Reuse one semantic stack across artifacts: embedder, vector store, graph, runtime memory, and learning memory should point at the same retrieval story.
6. Keep 8F trace lines concrete enough that a reviewer can see which local files informed the output.

## Anti-Pattern
1. Drafting from the handoff alone causes kind drift, because the requested filenames do not carry the builder field expectations by themselves.
2. Treating Knowledge Gluttony as decorative language instead of retrieval strategy leads to bloated prose with weak operational value.
3. Mixing memory-state content into knowledge artifacts blurs what should be searched as policy versus what should be recalled as state.
4. Writing all five artifacts independently without a shared semantic stack causes contradictions in dimensions, graph usage, and retrieval assumptions.

## Impact
1. Builder-first loading reduced the chance of missing required sections such as properties tables and boundary statements.
2. Local schema injection kept P01 and P10 responsibilities separable, which improves future retrieval and review.
3. Shared retrieval assumptions across artifacts improved coherence for future vector, graph, and memory integration.
4. Review effort shifted from structural correction to content calibration, which is the higher-value use of time for N04.

## Reproducibility
1. Conditions: the builder files, kind KCs, and nucleus contracts must exist and be readable before drafting.

4. Caveats: if the repository adds stronger validators, the same sequence still works but may require tighter byte budgeting.
