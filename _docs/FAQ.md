# CEX FAQ

> 10 most common questions answered.

---

### 1. What is a builder?

A builder is a factory blueprint that lives in `archetypes/builders/{kind}-builder/`.
Each builder contains 13 ISO files (manifest, system prompt, knowledge, instructions,
tools, output template, schema, examples, architecture, config, memory, quality gates,
collaboration). Builders define HOW to create a specific type of artifact.

### 2. What is a pillar?

A pillar is one of 12 organizational categories (P01-P12) that group related artifact
types. For example, P01 (Knowledge) holds knowledge cards, RAG sources, and glossary
entries. Each pillar has its own `_schema.yaml`, `_generator.md`, `templates/`, and
`examples/`.

### 3. What is a nucleus?

A nucleus is one of 8 domains (N00-N07) where real artifacts live. N00 (Genesis) is
the archetype nucleus -- it holds the builder definitions, pillar schemas, and shared
ISOs that all other nuclei inherit from. N01-N07 are business nuclei where domain
instances live. Example: `N02_marketing/P03_prompt/` holds marketing-specific prompts.

### 4. How do I add a custom kind?

Custom kinds go in the `_custom/` directory within a pillar. The 123 core kinds are
fixed and cannot be modified without architecture review. To add a custom kind:

1. Create `P{NN}_{pillar}/_custom/{your_type}/`
2. Add a `_schema.yaml` that inherits from the parent pillar schema
3. Create templates and examples following the parent pillar's conventions

### 5. How do I validate an artifact?

Three levels of validation:

```bash
# Structure check (folders, naming, schemas)
python _tools/cex_doctor.py

# Builder ISO completeness
python _tools/validate_builder.py archetypes/builders/{kind}-builder/

# Schema compliance for a specific file
python _tools/validate_schema.py
```

Manual checklist: frontmatter complete, density >= 0.8, naming follows pattern,
max 4KB, no prose > 3 lines, bullets max 80 chars.

### 6. How do I compile artifacts?

```bash
python _tools/cex_compile.py --all
```

This reads every `.md` artifact, extracts its YAML frontmatter, and generates a
compiled counterpart (`.yaml` or `.json`) in the `compiled/` directory. The compiled
format is defined by the `machine_format` field in the pillar's `_schema.yaml`.

### 7. Where are instances stored?

Instances (real artifacts, not templates) live in nuclei:

```
N{XX}_{domain}/P{NN}_{pillar}/{type}/my_artifact.md
```

Example: `N01_intelligence/P01_knowledge/knowledge_card/kc_competitor_analysis.md`

Templates live in `P{NN}_{pillar}/templates/`. Examples live in `P{NN}_{pillar}/examples/`.

### 8. What are the 8 LLM functions?

Every LLM system executes these functions in order (pipeline, not categories):

| # | Function    | Verb       | What it does                  |
|---|-------------|------------|-------------------------------|
| 1 | BECOME      | Be         | Set identity and persona      |
| 2 | INJECT      | Know       | Load context and knowledge    |
| 3 | REASON      | Think      | Chain-of-thought, planning    |
| 4 | CALL        | Do         | Use tools, APIs, MCPs         |
| 5 | PRODUCE     | Generate   | Create final output           |
| 6 | CONSTRAIN   | Restrict   | Apply schemas, validation     |
| 7 | GOVERN      | Evaluate   | Run quality gates and evals   |
| 8 | COLLABORATE | Coordinate | Handle signals and handoffs   |

Full spec: `LLM_PIPELINE.md` at the CEX root.

### 9. What is dual output?

Every artifact exists as two files:

- `.md` — human-readable source (you write this)
- `.yaml` or `.json` — machine-readable compiled output (generated by `cex_compile.py`)

The `.md` is the source of truth. The compiled file is for LLM consumption, embedding,
and retrieval. Never edit compiled files directly.

### 10. What is the quality gate?

Every artifact gets a quality score:

| Score | Tier     | What happens             |
|-------|----------|--------------------------|
| 9.5+  | Golden   | Reference quality        |
| 8.0+  | Skilled  | Published and indexed    |
| 7.0+  | Learning | Experimental, not public |
| < 7.0 | Rejected | Must be redone           |

Quality depends on: density (>= 0.8), completeness of frontmatter, specificity
(every sentence enables action without external docs), and adherence to schema
constraints.
